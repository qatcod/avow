# Forge v2.4 — Property/Metamorphic Test Generation — Design Spec

**Status:** Approved (brainstorming, 2026-06-26)
**Parent:** [Forge design spec](2026-06-26-forge-design.md) → v2 verification-hardening layer (the moat's layer I: "ground checks in execution, not judgment").

## Goal

Generate the strongest kind of test: ones that assert **laws holding for all inputs** (round-trips, idempotence, commutativity, ordering, algebraic invariants) and **fuzz thousands of random inputs** against them. No reference answer needed — assert "`decode(encode(x)) == x`" or "`add` is commutative", and let a fuzzer hunt for violations. The closest thing to *synthesizing a physics simulator* for software.

## The method

A model reads the goal and proposes **metamorphic relations / invariants**, emitted as **Hypothesis** property-based pytest files (`@given(...)` — Python's standard property/fuzz library generates inputs and shrinks failures, and runs through the existing Runner/pytest pipeline). Property tests **join the frozen suite as converge targets** — the builder must satisfy them, so Hypothesis fuzzes *during the build*, which is the entire point.

## Scope

**In (v2.4):** the generator (`forge/properties.py`), the `hypothesis` dependency, a loop hook that adds property tests to the visible frozen suite, and a standalone `forge propertize <goal> <out_dir>` CLI.

**Out (later):** reference-oracle-by-simplicity differential testing (a sibling of layer I — a separate sub-project); property tests as a *post-hoc* confidence signal (rejected — see below).

## Key decision: converge target, not post-hoc signal

Property tests are added to the frozen suite (the builder must pass them), **not** computed as a separate confidence signal. Rationale: fuzzing *during the build* catches bugs the example tests miss while there's still a builder to fix them — as a post-hoc signal you lose that. The bounded risk (a hallucinated *impossible* property could block convergence) is caught by the existing stop-conditions (plateau/budget → reports the unsatisfiable property), the mutation/intent signals, and the human gate. A wrong property is itself a suite-quality issue the rest of the moat surfaces.

## Components

`forge/properties.py`:

| Unit | Job |
|---|---|
| `generate_property_tests(goal, client, model, n) -> tuple[list[TestFile], int, int]` | LLM structured-output call → up to n Hypothesis property-test files (`TestFile` reused from `forge.examiner`) + token usage. `([], 0, 0)` when `n<=0` or `client is None`. |

The prompt instructs: import the impl + `from hypothesis import given, strategies as st`; use `@given(...)` with strategies matching the goal's input types; assert invariants/metamorphic relations true for *any* correct implementation (not specific output values unless a true invariant); bare `test_prop_*.py` filenames.

## Integration

- **Loop hook:** in `solve()`'s `write_tests` block, after the Examiner writes example tests and `split_suite`, IF `config.property_tests_enabled` and a `property_client` is provided: generate property tests, charge tokens, and **append them to the `visible` set** (frozen, not held out — they're acceptance criteria) before `_write_tests(frozen, visible)`.
- `solve()` gains a keyword-only `property_client=None` (parallel to `mutation_client`/`intent_client`); `None` → hook skipped (no offline mode), so offline runs stay runnable.
- `RunConfig` gains `property_tests_enabled: bool = True`, `property_model: str = "claude-opus-4-8"` (a capable model — getting invariants *right* matters), `property_tests_n: int = 4`.
- **`hypothesis`** added to `dependencies` (the generated frozen tests import it and run via the existing Runner's pytest, in the Forge venv).
- **Standalone:** `forge propertize <goal_file> <out_dir>` writes property tests for any goal — the "give me invariant tests for this spec" spin-out.
- Injectable client; the generator is fake-tested (no tokens).

## Honest framing

Property tests are execution-grounded (Hypothesis runs real inputs — ground truth for the slice it covers), but the *relations themselves* are LLM-proposed and could be wrong. That risk is the same class the rest of the moat addresses (mutation, intent, the human gate). The value is real: a single commutativity/round-trip property fuzzed over thousands of inputs catches whole classes of bugs that example tests miss.

## Testing strategy

- `generate_property_tests`: fake client returning a `_PropertySet` of `TestFile`s + usage; assert the goal is forwarded, the files flow through, tokens captured, the `n<=0`/`None`-client no-op, no sampling params.
- Loop hook: a fake `property_client` returning a *satisfiable* property (e.g. `add` commutativity) → the loop converges green with the property file present in `tests_frozen/`. Offline (the property test runs via the real Runner against the converged solution; Hypothesis is installed in the venv).
- `forge propertize` CLI: offline via a monkeypatched client → writes the files + prints them.

## Risks

- **Hallucinated/invalid property** — a syntactically-broken or logically-impossible property pollutes the suite. Mitigated by stop-conditions (the loop reports it couldn't converge), the back-translation/mutation signals, and the human gate; not eliminated.
- **`hypothesis` dependency** — one well-established library; the generated tests run through the existing pytest pipeline (no new runner).
- **Strategy mismatch** — the LLM may pick wrong `@given` strategies for the input types; a mismatched strategy errors at collection/run and is treated as a failing test (the builder/loop surfaces it).
