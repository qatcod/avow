# Hermit v2.1 — Mutation Testing (suite-strength scoring) — Design Spec

**Status:** Approved (brainstorming, 2026-06-26)
**Parent:** [Hermit design spec](2026-06-26-hermit-design.md) — this is the first sub-project of the v2 "verification hardening" layer (the moat).

## Goal

Given a solution and its frozen test suite, deliberately break the solution in many small ways and measure what fraction of those breakages the suite *catches*. **Kill rate = suite strength.** Survivors pinpoint exactly where the suite is blind.

## Why this is the cornerstone of v2

It's the single highest-value verification signal: fully **execution-grounded** (no extra LLM judgment — a mutant is killed iff the suite fails on it), self-contained, reuses the existing v1 `Runner`, and it *is* the sellable spin-out — `hermit mutate <repo> <tests>` produces a mutation-tested suite report for *any* codebase, sidestepping the crowded "autonomous coder" market.

## Scope

**In (v2.1):**
- The mutation **engine** (`hermit/mutation.py`).
- A standalone **`hermit mutate <solution_dir> <tests_dir>` CLI** — the headline deliverable; runs on any code, not just a Hermit run.
- A minimal **loop hook**: after `solve()` converges green, run mutation testing on `best_dir`, record the score + survivors in the run log and `SolveResult`.

**Out (later sub-projects):**
- Auto-strengthening the suite by feeding survivors back to the Examiner (v2.x).
- Folding the mutation score into a calibrated confidence number (v2.3).
- Parallel mutant execution (v2.x perf).

## The honest position (calibrated, not certain)

- Some mutants are **semantically equivalent** to the original (e.g. a constant change with no behavioral effect) and can never be killed — they appear as false survivors.
- Some mutants **trivially error** (import/syntax-adjacent) and are killed without meaningfully exercising the suite.

So the kill rate is a **calibrated signal, not a verdict**. Survivors are reported as "potential gaps to inspect," and the report says so. This matches the rest of Hermit's "trustworthy because it knows what it doesn't know" ethos.

## Mutant generation — hybrid (AST + a few LLM)

Two generators, unified behind one `Mutant(source, description, origin)` type so they compose:

- **AST mutators (deterministic, free, the core):** walk the solution's AST and apply **one mutation per mutant**, re-emitting source via `ast.unparse`. Operator set: arithmetic (`+ - * / %`), comparison (`== != < <= > >=`), boolean (`and ↔ or`), constants (numeric ±1, `True ↔ False`, empty↔nonempty string), and `return <expr>` → `return None`. One mutation per mutant means a survivor names exactly one gap. Reproducible (great for TDD).
- **LLM mutators (a few, cross-model, for depth):** a structured-output call asks a *different* model than the Builder for `n` "realistic bug" variants (plausible mistakes a developer would make) that a correct suite should catch. Catches subtler logic gaps AST can't. Injectable client (fake in tests, like the Examiner); token cost surfaced on the result and charged to the Budget by the loop hook.

## Components (one module, one CLI verb, one hook)

`hermit/mutation.py`:

| Unit | Responsibility |
|---|---|
| `Mutant(source: str, description: str, origin: str)` | one mutated source + what changed (`"BinOp Add->Sub"`) + origin `"ast"`/`"llm"` |
| `ast_mutants(source: str) -> list[Mutant]` | deterministic AST operator mutants, one mutation each |
| `llm_mutants(source, goal, client, model, n) -> tuple[list[Mutant], int, int]` | n realistic-bug variants + (input_tokens, output_tokens) |
| `Survivor(file: str, description: str, origin: str)` | a mutant the suite failed to catch |
| `MutationResult(total, killed, survived, score, survivors, llm_input_tokens, llm_output_tokens)` | `score = killed/total` (1.0 if `total==0`, flagged vacuous) |
| `run_mutation_testing(solution_dir, frozen_tests, test_command, *, max_ast_mutants, llm_n, timeout, client, model, goal) -> MutationResult` | generate mutants per top-level module → for each, copy the solution to a temp dir, overwrite the one module with the mutant, **grade via the v1 `Runner`** → **killed = `not result.is_green`** |

Plus:
- **CLI:** `hermit mutate <solution_dir> <tests_dir> [--config hermit.yaml]` → prints the score, killed/total, and the survivor list (AST-only by default offline; `--llm` opts into hybrid). The spin-out.
- **Loop hook:** in `solve()`, after a green converge, call `run_mutation_testing(best_dir, frozen, ...)`, charge the LLM tokens to the Budget, record `mutation_score` + survivor count in the run log and on `SolveResult`.

## Integration with v1 (consumer, not surgery)

- **Zero changes to the Runner.** Mutation testing is a *consumer*: for each mutant it grades the mutated solution copy against the frozen tests in the Runner's existing ephemeral-copy flow. `killed = not result.is_green` (suite failed or errored → caught it).
- **`SolveResult`** gains optional `mutation_score: float | None` and `survivors: int` (None when the run didn't reach green / mutation was skipped).
- **`RunConfig`** gains `max_ast_mutants: int = 50`, `llm_mutants_n: int = 3`, `mutation_model: str = "claude-sonnet-4-6"` (cross-model from the default Opus Builder), and `mutation_enabled: bool = True` (the post-converge hook).
- **LLM mutants are injectable** (fake client in tests); their cost is charged to the Budget by the loop hook (the engine just reports the token counts).

## Defaults (veto any)

- `max_ast_mutants=50` (cap — mutation count can explode), `llm_mutants_n=3`, `mutation_model` = the examiner/cross model.
- **Killed = suite not green** (fail *or* error). Equivalent-mutant + trivial-error noise reported honestly, not hidden.
- **Serial mutant execution** for v2.1 (parallelize later); bounded by `max_ast_mutants` + the per-run timeout.
- `hermit mutate` defaults to **AST-only** (offline, free, deterministic); `--llm` adds the cross-model mutants.

## Testing strategy

- `ast_mutants`: known source → known mutants (e.g. `return a + b` yields a `return a - b` mutant); deterministic count.
- `run_mutation_testing`: a real tiny solution + a **strong** suite (kills all behavioral mutants → high score) vs a **weak** suite (lets a mutant survive → score < 1.0 with a named survivor). All via the real Runner (fast, no network).
- `llm_mutants` + hybrid wiring: a fake client returning canned mutants + token usage; assert mutants flow through and tokens are reported.
- CLI + loop hook: offline (AST-only / fake builder), assert score is reported and recorded.

## Risks

- **Mutation runtime** = O(mutants × test-time); the `max_ast_mutants` cap + per-run timeout bound it, but a large solution with a slow suite is expensive — documented, parallelization deferred.
- **Equivalent mutants** inflate the survivor list with false gaps — reported as "potential," never as guaranteed weaknesses.
- **LLM mutants can be trivially broken** (syntax-adjacent) → killed cheaply without testing the suite; acceptable noise, AST is the rigorous core.
