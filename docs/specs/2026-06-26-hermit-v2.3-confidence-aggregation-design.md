# Avow v2.3 — Calibrated Confidence Aggregation — Design Spec

**Status:** Approved (brainstorming, 2026-06-26)
**Parent:** [Avow design spec](2026-06-26-avow-design.md) → v2 verification-hardening layer. The **capstone** that combines [v2.1 mutation](2026-06-26-avow-v2.1-mutation-testing-design.md) + [v2.2 intent](2026-06-26-avow-v2.2-back-translation-design.md) + v1 hold-out into one trust number.

## Goal

Combine the verification signals into a single **calibrated confidence** for a green solution, and rewire "done" so a solution that *passes the tests* but scores low on trust gets **flagged and escalated**, not silently declared finished. This is the piece that makes "tests pass" ≠ "done."

## The signals aggregated (all already produced, all in [0,1])

- **hold-out pass rate** — did it overfit the visible tests?
- **mutation score** — are the tests rigorous enough to catch bugs? (v2.1)
- **intent match** — do the tests test the *right* goal? (v2.2)

Behavioral-green is the **precondition** (confidence is only computed for a green solution), not a variable in the average — it's always 1.0 post-converge.

## Aggregation — weighted average over present signals

A **pure-math** aggregator (no LLM, deterministic, fully unit-testable): weighted average of the signals that are **present**, renormalizing the weights when a signal is absent (e.g. no hold-out configured, or no intent client). Reported with the **per-signal breakdown** so a human sees *why* the number is what it is — transparency over a single opaque number. (Rejected: min/product — one noisy signal shouldn't tank the whole verdict.)

## Components

`avow/confidence.py` — pure, no LLM:

| Unit | Job |
|---|---|
| `ConfidenceResult(score: float, breakdown: dict[str, float], weights_used: dict[str, float])` | the number + per-signal inputs + the renormalized weights actually applied |
| `aggregate_confidence(signals: dict[str, float \| None], weights: dict[str, float]) -> ConfidenceResult` | filter to present signals with positive weight; renormalize their weights; weighted average; empty inputs → `score=0.0, breakdown={}, weights_used={}` (caller treats empty as "not computable") |

Plus the loop rewiring and a standalone **`avow verify <solution> <tests> <goal>`** CLI that runs mutation + intent on any artifact and reports the aggregated confidence — the "how much should I trust this?" spin-out.

## How "done" gets rewired (the meat)

On a green converge, the loop collects the hold-out **score** + mutation score (+ the intent score from the early hook), aggregates → `confidence`, then:

- **confidence ≥ threshold** (or gating off, or confidence not computable) → `success`, reason `"green"`.
- **confidence < threshold** → reason `"low_confidence"`, the full breakdown surfaced; if an `escalate` callback is wired, it's called with the breakdown so a human can **accept-anyway** (→ `success`, reason `"green_human_override"`) or reject (→ `success=False`, reason `"low_confidence"`). `best_dir` always holds the green-but-untrusted solution for inspection.

**Behavior change (approved):** v1's binary `overfit_on_holdout` gate is **replaced** — hold-out becomes a *score* feeding confidence, so a partial hold-out failure *lowers confidence* rather than hard-failing. `_holdout_green` → `_holdout_score` (returns the pass fraction; 1.0 when no hold-out is configured).

## Integration

- `SolveResult` gains `confidence: float | None = None` and `confidence_breakdown: dict = {}`.
- `solve()` gains a keyword-only `escalate=None` (called only on low confidence when gating is on).
- `RunConfig` gains `confidence_threshold: float = 0.7`, `confidence_weights: dict[str, float]` (default equal over `holdout`/`mutation`/`intent`), and `confidence_gating: bool = True` (toggle — `False` keeps it report-only, never flipping `success`).
- The confidence + breakdown are recorded on the run log.

## Backward compatibility

Every existing v1/v2.1/v2.2 loop test produces high confidence (mutation 1.0, hold-out 1.0, intent ≥ 0.83 → confidence ≥ 0.94 ≥ 0.7), so they still end `success` with reason `"green"`. The new low-confidence / escalation behavior is covered by new tests. The new `SolveResult` fields are trailing with defaults.

## Honest framing

Confidence is a **calibrated, transparent** number, not certainty — its inputs are themselves signals (two of them LLM-judged). The breakdown is always surfaced so the verdict is auditable. A `low_confidence` result is the system **telling you it doesn't trust its own green** — the most valuable thing the moat produces.

## Defaults (veto any — locked at approval)

- Weighted average over present signals, renormalized, reported with breakdown.
- Done = green ∧ confidence ≥ threshold (gating on, toggleable); low → escalate.
- Fold hold-out into the signals (replace the binary gate).
- `confidence_threshold = 0.7`; equal default weights.

## Testing strategy

- `aggregate_confidence`: pure unit tests — equal weights, custom weights, a missing signal (renormalization), all-missing (empty result), single signal.
- Loop: high-confidence → `success`/`"green"`; low-confidence (controlled via a fake intent score) without escalate → `success=False`/`"low_confidence"`; low-confidence with `escalate` accepting → `success`/`"green_human_override"`; `confidence_gating=False` → low confidence still `success`. All offline (fake intent client, AST-only mutation).
- `avow verify` CLI: offline via a monkeypatched client → prints confidence + breakdown.

## Out of scope (later)

- Learned/calibrated weights from outcome data (v3+) — v2.3 uses fixed configurable weights.
- A confidence signal from a cross-model panel (v2.5) — the panel is a separate sub-project; when it lands it becomes another signal in the same aggregator.
