# Avow v2.2 — Back-Translation Intent Check — Design Spec

**Status:** Approved (brainstorming, 2026-06-26)
**Parent:** [Avow design spec](2026-06-26-avow-design.md) → v2 verification-hardening layer. Sibling of [v2.1 mutation testing](2026-06-26-avow-v2.1-mutation-testing-design.md).

## Goal

Catch the failure mode mutation testing **cannot**: the Examiner writing a rigorous suite for the **wrong goal**. A different model reads the test suite *blind* and reconstructs the goal it thinks the suite is testing; we compare that to the real goal. Divergence = the suite drifted from intent — caught *before* the build loop burns budget building to the wrong target.

## The method — two-step and blind (the load-bearing choice)

- **Step 1 — blind back-translate:** a model reads **only the test files** (never the goal) and writes its best inference of the goal. Blindness is what makes this rigorous: if the suite doesn't actually encode the goal, the inferred goal *will* diverge.
- **Step 2 — judge:** a model reads the **original goal + the inferred goal** → an `intent_score` (0–1, 1=perfect match) + concrete `divergences` (requirements the suite missed or invented).

Rejected alternative: a single call that sees *both* goal and suite ("does this test that?"). It lets the model anchor on the goal and rationalize, so it never tests whether the suite *alone* conveys the intent.

## Scope

**In (v2.2):** the engine (`avow/backtranslation.py`), a standalone **`avow intent-check <goal.md> <tests_dir>` CLI** (spin-out: "does my suite actually test what I think?"), and an **early loop hook** that records the intent score right after the suite is written.

**Out (later):** auto-escalation / blocking on a low score (belongs to **v2.3 confidence aggregation**); the v2.2 hook is **report-only**.

## Honest framing

The intent score is an **LLM judgment** (two LLM calls), not ground truth — a **calibrated signal**, not a verdict. Cross-model decorrelation (back-translator ≠ Examiner) reduces correlated blind spots but doesn't eliminate them. Always reported with its divergences so a human can see *why* it diverged.

## Components (mirrors the v2.1 shape)

`avow/backtranslation.py`:

| Unit | Job |
|---|---|
| `_InferredGoal(inferred_goal: str)` | internal structured-output model for step 1 |
| `back_translate(test_sources: str, client, model) -> tuple[str, int, int]` | blind: suite text → `(inferred_goal, in_tok, out_tok)` |
| `IntentMatch(score: float, divergences: list[str])` | structured judge output |
| `judge_intent_match(original_goal, inferred_goal, client, model) -> tuple[IntentMatch, int, int]` | compare → score + divergences |
| `IntentResult(score, inferred_goal, divergences, input_tokens, output_tokens)` | aggregate result |
| `run_intent_check(goal, frozen_tests_dir, client, model) -> IntentResult` | read `test_*.py` from the dir → back_translate → judge → aggregate (sums tokens across both calls) |

Both LLM calls use the `anthropic` structured-outputs path (`messages.parse(..., output_format=...)` → `.parsed_output`, `.usage`); never set `temperature`/`top_p`/`top_k`. The `client` is injectable (fake in tests; the fake dispatches on `output_format` to return an `_InferredGoal` then an `IntentMatch`).

## Integration

- **Early loop hook:** in `solve()`, after the Examiner writes the frozen suite (and the optional human approval gate passes), **before** the build loop, run `run_intent_check(goal, frozen, intent_client, config.backtranslation_model)` *only when* an `intent_client` is provided and `config.intent_check_enabled`. Charge the tokens to the Budget, record `intent_score` + divergences on the run log, and surface `intent_score` on `SolveResult`. Report-only.
- **Standalone CLI:** `avow intent-check <goal_file> <tests_dir>` → prints the inferred goal, the match score, and the divergences. Requires credentials (it's inherently LLM-based — no deterministic fallback, unlike mutation's AST core).
- `RunConfig` gains `backtranslation_model: str = "claude-opus-4-8"` (≠ the Sonnet Examiner, for decorrelation) and `intent_check_enabled: bool = True`.
- `SolveResult` gains `intent_score: float | None = None`.
- `solve()` gains a keyword-only `intent_client=None` (parallel to `mutation_client`). `None` → the intent hook is skipped (back-translation has no offline mode), so the loop stays runnable without credentials; the hook is exercised in tests via a fake `intent_client`.

## Defaults (veto any)

- Two-step blind; early hook; report-only; `backtranslation_model = claude-opus-4-8`.
- `intent_check_enabled = True`, but the hook only fires when an `intent_client` is passed (so `avow solve` does an intent check only when wired with a client; offline/test runs skip it cleanly).

## Testing strategy

- `back_translate` / `judge_intent_match`: fake client returning canned structured output + usage; assert the suite text / goals are forwarded into the prompts, the parsed result flows through, tokens captured, no sampling params.
- `run_intent_check`: a fake client that dispatches on `output_format` (returns `_InferredGoal` for step 1, `IntentMatch` for step 2); reads `test_*.py` from a temp dir; asserts the aggregated score + summed tokens.
- CLI + loop hook: fake client; assert the score is printed / recorded on `SolveResult`.

## Risks

- **Two LLMs agreeing ≠ ground truth** — the residual correlated-blind-spot risk is mitigated (not removed) by cross-model; surfaced via divergences and the honest "signal not verdict" framing.
- **A vague goal.md** makes the judge's comparison noisy — out of scope to fix here; the divergences list still gives a human something concrete to read.
- **Cost** — two LLM calls per run; cheap relative to the build loop, charged to the Budget, gated by `intent_check_enabled` + the presence of an `intent_client`.
